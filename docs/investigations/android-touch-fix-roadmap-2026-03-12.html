What Was Already Proven

  • Mouse input works on Android, so the shared lyric interaction model is not the only possible runtime path.
  • The first full vs off experiment did not restore clean touch parity.
  • That does not clear the browser layer, because browser-side blockers still remained active in both APKs.

What Still Blocks Touch

  • Static viewport lock with user-scalable=no
  • #lyric-display { touch-action: none; }
  • Android-wrapper lyric-display overflow suppression
  • Browser gesturestart prevention

Current Experiment

Add a second profile for the wrapped browser itself:

  • AXOLYNC_ANDROID_BROWSER_TOUCH_SUPPRESSION_PROFILE
  • passed into browser build as VITE_AXOLYNC_ANDROID_TOUCH_SUPPRESSION_PROFILE
  • full keeps browser-side suppression on
  • relaxed disables browser-side suppression overrides for investigation

Exit Criteria

  • If browser-side relaxed materially improves touch, narrow that layer piece by piece.
  • If behavior is still chaotic, stop spending time on suppression toggles and move to target-layer hit testing and event-path tracing.
  • Only after that should the permanent control-panel fallback be positioned as either essential or secondary.
